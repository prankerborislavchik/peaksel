### [ Сценарий 1 - Регистрация пользователя ]

1. Пользователь переходит по ссылке "Войти" в шапке страницы и попадает на страницу авторизации (входа)
2. Оттуда переходит по ссылке "Зарегистрироваться", если у него ещё нет аккаунта и видит перед собой форму из 4х обязательных для заполнения полей: Имя пользователя, Почта, Пароль, Повторите пароль
3. Пользователь вводит имя. Имя должно быть от 2ух до 24ёх символов.  
4. Пользователь вводит свою почту, которая будет использоваться. Длина пользовательской почты должна быть до 254 символов. Также валидность почты проверяется регулярным выражением.
5. Пользователь вводит пароль и повторяет его. Пароль обязательно должен быть заполнен и должен содержать не более 64 символов
6. Пользователь нажимает на кнопку "Зарегистрироваться".
7. Если пользователь неправильно заполнил какое-либо поле, или если пароли не совпадают друг с другом, то под соответствующими полями появляется сообщение об ошибке
8. Если пользователь прошёл валидацию, то данные отправляются на сервер и обрабатываются следующим образом:
9. Если пользователь с данной почтой уже зарегистрирован в системе, то запрос возвращает ошибку с сообщением: "Пользователь с данным email уже зарегистрирован"
10. Если пользователь с данной почтой не зарегистрирован, то в системе создаётся пользователь вместе с токеном для активации профиля и корзиной. 
11. Система отправляет на указанную почту письмо с подтверждением регистрации
12. Далее пользователя перенаправляют на страницу, где его информируют, что для подтверждения регистрации необходимо проверить почту и перейти по ссылке из сообщения.
13. После перехода по ссылке, пользователь считается авторизованным и его аккаунт является подтверждённым.

### [ Сценарий 2 - Авторизация пользователя]
1. Пользователь переходит по ссылке "Войти" в шапке страницы и попадает на страницу авторизации (входа), где видит перед собой форму из двух полей: Почта пользователя, Пароль. 
2. Пользователь вводит свою почту. Первично почта проверяется по следующим критериям: соответствует регулярному выражению почты, и её длина до 254 символов.
3. Далее пользователь вводит пароль. Пароль обязательно должен быть заполнен и должен содержать не более 64 символов
4. Пользователь нажимает на кнопку "Войти".
5. Если пользователь неправильно заполнил какое-либо поле, то под соответствующими полями появляется сообщение об ошибке
6. Если пользователь прошёл валидацию, то данные отправляются на сервер и обрабатываются следующим образом:
7. Если пользователь с данным email не зарегистрирован, то на клиент возвращается сообщение об ошибке.
8. Если у пользователя не подтверждён аккаунт, то ему повторно отправляется письмо для подтверждения регистрации и на клиент возвращается ошибка 
9. Далее сравнивается, соответствует ли введённый пользователем пароль с паролем, указанным при регистрации.
10. Если пароли не совпали, то на клиент возвращается ошибка
11. Если в пунктах 7-10 не возникло ошибки, то пользователь считается авторизованным

### [ Сценарий 3 - Восстановление пароля пользователя]
1. Пользователь переходит по ссылке "Войти" в шапке страницы и попадает на страницу авторизации (входа).
2. Оттуда переходит по ссылке "Восстановить", где видит перед собой форму содержащую единственное поле: Почта
3. Пользователь вводит свою почту. Первично почта проверяется по следующим критериям: соответствует регулярному выражению почты, и её длина до 254 символов.
4. После ввода почты пользователь жмёт "Восстановить"
5. Если пользователь прошёл валидацию, то данные отправляются на сервер и обрабатываются следующим образом:
6. Если пользователь с указанной почтой не зарегистрирован в системе, то на клиент возвращается ошибка
7. Если у пользователя уже есть прикреплённый в БД токен для восстановления пароля, который ещё не истёк, то пользователь по новой получает письмо с уже имеющимся кодом для восстановления пароля
8. Если у пользователя ещё нет прикреплённого не истёкшего токена, то к пользователю прикрепляется токен, устанавливается срок, после которого будет считаться истёкшим (2 часа), после чего этот токен отправляется на почту пользователя в виде ссылке.
9. После чего на клиент возвращается успешный результат (если до этого не вернулась ошибка), оповещающий пользователя, о том, что для восстановления пароля необходимо проверить почту и перейти по ссылке в письме.
10. Пользователь проверяет почту, переходит по ссылке в письме (токен для восстановления лежит в ссылке из письма)
11. Если в БД нет ни одного пользователя с токеном для восстановления пароля из ссылки, то на странице выводится сообщение о недействительности ссылки.
12. Если срок действия токена истёк, пользователю на странице выводится соответствующее сообщение
13. Если же в п. 11-12 не возникло проблем, то пользователь видит перед собой форму с 2умя полями: Пароль, повторите пароль
14. Пользователь вводит новый пароль и повторяет его. Пароль обязательно должен быть заполнен и должен содержать не более 64 символов. 
15. Пользователь нажимает кнопку "Восстановить"
16. Если пароли не совпадают, пользователь сразу увидит ошибку, если пароли совпадают, то данные отправляются на сервер
17. На сервере проходит минимальная валидация, и пользователю устанавливается новый пароль, меняется время последнего изменения пароля, и зачищается информация по токену для смены пароля и сроку его истечения, после чего на клиент возвращается ответ, после которого пользователь может войти в свой аккаунт под новым паролем  

### [ Сценарий 4 - Заказ звонка]
1. Пользователь нажимает по кнопке "Заказать звонок" (или на иконку звонка на мобильной версии) и видит перед собой форму с одним полем ввода: вводом самого номера
2. Пользователь вводит номер (вводит только цифры, остальные знаки проставляет маска). Вводятся только Российские номера (+7, 8).
3. Также есть возможность вставить номер, маска также подгонит номер под формат (Если что, маска преобразовывает входящие числа (и 2 и 3 пункт) в подобный формат: +7 (000) 000-00-00, либо же: 8 (000) 000-00-00)
4. Далее пользователь жмёт кнопку "Отправить"
5. Если номер неполный, то выводится сообщение об ошибке, иначе данные отправляются на сервер
6. Если пользователь авторизован, то номер привязывается к профилю пользователя
7. Если пользователь не авторизован, то номер попадает в базу неотслеженных номеров телефона 
8. На клиент возвращается сообщение, что пользователю перезвонят
9. Пользователь ждёт звонка 

### [ Сценарий 5 - Обратная связь]
1. Пользователь переходит по ссылке "Связь с нами" в шапке страницы и попадает на страницу с формой, состоящей из 2ух полей: Ваши данные, Введите сообщение.
2. Пользователь вводит свои данные. Данные пользователя обязательны для заполнения. Длина данных пользователя в строке должна содержать до 120 символов
3. Пользователь оставляет своё обращение в поле "Введите сообщение". Данное поле обязательно для заполнения и должно содержать от 2ух до 1000 символов
4. Пользователь жмёт кнопку "Связаться с нами". Если пользователь неправильно заполнил какое-либо поле, то под соответствующими полями появляется сообщение об ошибке. 
5. Если поля заполнены корректно, то данные отправляются на сервер
6. Обращение пользователя записывается в БД
7. Сервер возвращает на клиент успешный ответ
8. Клиент ждёт обратной связи по своему обращению

### [ Сценарий 6 - Поиск телевизора по фильтрам ]
1. (Пользователь на главной странице) Перед пользователем для удобного поиска телевизора имеется следующее: Сами фильтры, постраничный вывод (пагинация) и список самих телевизоров (20 телевизоров на странице по умолчанию). Также имеется кнопка для переключения режима просмотра телевизоров (Телевизоры могут располагаться по одному в ряду, либо по несколько в ряд (в зависимости от разрешения экрана от 2ух до 4ух в ряд). !Чисто UI история, не имеет отношения к запросам на сервер!)
2. Пользователь выбирает опции фильтров из предложенного перечня фильтров (на мобильной версии необходимо открыть фильтры, нажав на соответствующую кнопку)
3. После выбора фильтров, пользователь жмёт кнопку применить. 
4. Данные пагинации на странице сбрасываются. Данные отправляются на сервер
4. На сервере формируется запрос к БД из тех характеристик, которые имеются у модели Телевизора.
5. Сервер возвращает на клиент сами телевизоры, количество телевизоров, соответствующие критериям, номер страницы и количество телевизоров на странице (ещё возвращается сдвиг, но данная характеристики не используются на клиенте, т.к. есть номер страницы)
6. Далее на клиенте отрисовывается новая страница с данными от сервера
7. Пользователь может перемещаться по страницам при помощи пагинации, присутствующей при количестве страниц более одной (количество страниц = общее количество телевизоров, соответствующих характеристикам : размер страницы). При перемещении по страницам фильтры не сбрасываются (!Но при изменении фильтров, сбрасываются страницы, т.е. при нажатии на кнопки "Применить" и "сбросить" сначала отобразится первая страница!)
8. Также присутствует кнопка "Сбросить", которая по нажатию сбрасывает все фильтры и возвращает пользователя на первую страницу (без фильтров соответственно)
9. Также присутствует удобная страница для каждого телевизора по отдельности. Пользователь может перейти на страницу заинтересовавшего его телевизора и более подробно изучить его характеристики

В зависимости от расширения экрана и от мода просмотра (на главной странице, в зависимости от состояния кнопки отображается либо 1, либо несколько телевизоров в один ряд) у кнопки будет следующее содержимое (!Если телевизор ещё не лежит в корзине!): Добавить в корзину, либо же иконка самой корзины соответственно.

7. А теперь можно сказать, что будет с кнопкой, при условии того, что телевизор уже в корзине:
8. На самой кнопке в зависимости от расширения экрана и от мода просмотра (на главной странице, в зависимости от состояния кнопки отображается либо 1, либо несколько телевизоров в один ряд) у кнопки будет следующее содержимое (!Если телевизор лежит в корзине!): Уже в корзине, либо же иконка галочки. Также кнопка будет иметь отличный цвет.
9. При нажатии на данную кнопку (рассматривается только в случае если пользователь авторизован, т.к. действия при нажатии у неавторизованного пользователя уже прописаны в п.4) пользователь будет переходить на страницу корзины

### [ Сценарий 8 -  Добавление телевизора в каталог (функция из Панели Администратора)]
1. Первый обязательный пункт в функциях из Панели администратора: Все маршруты /admin являются защищёнными, т.е. доступными либо администратору, либо владельцу, ну никак не простому пользователю. Перейти на панель администратора можно при помощи ссылки "Панель администратора", находящейся в шапке страницы. Данная ссылка видна только администраторам и владельцу. Обычному пользователю ссылка не видна, но даже если пользователь попробует самостоятельно в браузере перейти на страницу администратора (вбив в строку браузера любой маршрут начинающийся с /admin), то увидит он только индикатор загрузки, после которого он будет перенаправлен на страницу авторизации. В общем, админ панель доступна только администраторам и владельцу. !Дальнейшие действия рассматриваются с условием того, что пользователь уже находится на страницы самой админ панели и является администратором или основателем!
2. Для добавления телевизора в каталог, администратор/владелец переходит по ссылке "Добавить телевизор" на панели администратора
3. Перед пользователем открывается страница с большой формой, в которую необходимо заполнять характеристики телевизора (Для характеристик отдельно прописана валидация и различные опции. Также могут отличаться и сами характеристики. Так, например, разрешением может быть только одно из нескольких предложенных значений, а беспроводной связи может быть несколько значений из предложенного списка, где-то администратор или владелец вводит значения самостоятельно и т.п.)
4. После заполнения полей, которые получилось заполнить администратор/основатель жмёт на кнопку "Создать", после нажатия на которую запускается прописанная валидация
5. Если какие-то пункты не прошли валидацию, то выдаётся ошибка и под соответствующими полями показывается сообщение с самой ошибкой
6. Если валидация пройдена, то данные отправляются на сервер
7. На сервере данные также проверяются, а также проверяется, отправлен ли запрос админом/владельцем (если запрос отправлен не админом/владельцем, возвращается ошибка) (и ещё небольшой ряд проверок)
8. После проверок создаётся экземпляр телевизора, который добавляется в БД, после чего администратору/владельцу возвращается успешный ответ
9. Теперь телевизор доступен на главной странице, доступен при поиске и т.п.
